# Project Instructions

## Communication Style

- Explain changes as if the user is non-technical by default.
- You may use technical words when useful, but define them in plain English right away.
- Prefer step-by-step explanations.
- Use analogies when they make the explanation easier to understand.
- Avoid assuming the user knows web-development or infrastructure jargon.
- When describing risk, explain both what could go wrong and what the practical effect would be.

## Product Context

- This is a personal tool for one user, not a multi-tenant SaaS product.
- Prefer simple, low-maintenance solutions over enterprise-style architecture.
- Local persistence is acceptable when it reduces complexity and still keeps personal data reasonably safe.
